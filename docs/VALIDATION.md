# Validation

This document summarizes the validation strategy for the first public release of `RLPlasticity`.

## Goal

The goal of validation is not to prove that every diagnosis is universally correct. The goal is to show that the tool behaves sensibly on:

- a healthy baseline
- encoder-side failure
- trunk-side failure
- head-side failure
- global plasticity stall
- checkpoint history generation

## Validation Cases

Generated artifacts live in [docs/validation](validation).

Cases currently included:

- `healthy`
  - expected: no acute issue
- `frozen_encoder`
  - expected: `encoder_bottleneck`
- `frozen_policy_head`
  - expected: `head_saturation`
- `frozen_trunk`
  - expected: `trunk_bottleneck`
- `global_stall`
  - expected: `global_plasticity_stall`
- `checkpoint_sequence`
  - expected: history rows and trend-aware metrics are present

The validation suite is generated by:

```bash
python -m examples.validation_suite --output-dir docs/validation
```

## Why These Cases Matter

These cases correspond to common triage scenarios in RL debugging:

- healthy baseline
  - confirms the tool can stay quiet when no obvious issue is present
- frozen encoder
  - checks whether the tool can separate encoder-side failure from downstream updates
- frozen policy head
  - checks whether the tool can identify head-side bottlenecks
- frozen trunk
  - checks whether the tool can identify shared-body bottlenecks rather than only encoder or head failures
- global stall
  - checks whether the tool can detect low-gradient, low-update collapse
- checkpoint sequence
  - checks whether history metadata survives aggregation and can support trend analysis

## Known Limits

- Validation is still synthetic and small-scale.
- Rules are heuristic, not learned or formally calibrated.
- More architecture families and RL algorithms should be added in future releases.
